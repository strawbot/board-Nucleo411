🌿 Create Feature Branch

Naming
feat/<ticket>-short-desc
Base
Always branch from latest main or develop
Lifetime
Short-lived — merge within days, not weeks
Tip
Rebase frequently to avoid large conflicts

💾 Commit Best Practices

Size
One logical change per commit
Message
Conventional: feat(scope): description
Body
Explain "why", not just "what"
Avoid
Committing secrets, large binaries, or debug code

🔍 Self Review Checklist

Diff
Read your own diff as if you were the reviewer
Tests
Run the full test suite locally before pushing
Lint
Zero warnings before opening a PR
Scope
Remove unrelated changes or open a separate PR

📬 Opening a Pull Request

Template
Fill out PR template completely
Description
What changed, why, how to test
Labels
bug / feat / chore / breaking-change
Size
Aim for <400 line changes per PR

⚙️ Automated Checks

Build
Compile / bundle must succeed
Tests
Unit + integration, coverage threshold
Lint
ESLint / Ruff / Checkstyle — zero errors
Security

Correctness

Security

💬 Leaving Feedback

Tone

Severity
Label: Blocker / Critical / Major / Minor / Nit
Suggestions

Praise

Level Definition Examples Required action Merge allowed?
🛑 Blocker Correctness or security bug that will cause failures or vulnerabilities in production SQL injection data loss bug broken auth race condition Must be fixed before merge. Reviewer re-reviews the fix. No
🔴 Critical Serious issue that significantly degrades quality, performance, or reliability N+1 query missing error handling memory leak Must be fixed before merge or tracked in a follow-up issue with a deadline. No*
🔵 Major Meaningful problem worth fixing now, but won't cause immediate production issues missing tests unclear naming poor abstraction Author should address. If not fixed, must leave a comment explaining why. With comment
🟢 Minor Small improvement that would make the code better but isn't required variable name comment wording minor refactor Author's discretion. Can address in a follow-up PR. Yes
⬜ Nit Tiny style preference or cosmetic suggestion — optional to address whitespace trailing comma word choice Prefix comment with "nit:" — author can ignore with no explanation needed. Yes

💡 Comment Labelling Convention

blocker: This must be fixed.
critical: Very important issue.
major: Should be addressed.
minor: Nice to fix but optional.
nit: Tiny preference, ignore freely.
question: I don't understand this — clarify?
suggestion: Consider this alternative.
praise: This is great, well done! 🎉
